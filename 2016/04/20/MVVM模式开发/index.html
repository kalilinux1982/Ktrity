针对文章和项目的分析摘录一下：

  • 一个 Activity 或 Fragment（一般都 with XML） 对应一个 ViewModel，对于一个基础 View（XML）可以通过继承对应的 ViewModel 实现重用，本文的代码也有体现
  • 绑定操作都放在 ViewModel 里，然后 ViewModel 里可以有多个 domain 中的 Interator(UseCase) 来得到 View 需要渲染的数据 Model
  • 关于评论中的

分别建立了三层模型类，分别以User UserEntity UserModel 这样的名字来区分层.

作者也给出对应的说法吧：

最核心的 databinding 还不完善，基本只能往 xml 里绑，现在先看看就好。 另外 Fernando Cejas(android10) 的 Android-CleanArchitecture 本身就是把展现层、业务层和数据层搞得往复杂化的，很多时候就是直接服务层的一个类再调用数据层的一个类就行了，另外像他每层还搞个自己的 Model Class 是很没有必要的,之前也在 Issue 里讨论过 issues72 等等

个人觉得是让每个层进行分离，互不干扰，利于做单元测试。

关于这篇文章：# 从零开始的Android新项目3 - MVPVM in Action, 谁告诉你MVP和MVVM是互斥的的摘录（需要关注底下评论中的讨论）：

对于MVVM通过View和ViewModel的双向绑定，让我们可以

  • 直接更新ViewModel，View会进行对应刷新
  • View的事件直接传递到ViewModel，ViewModel去对Model进行操作并接受更新。
