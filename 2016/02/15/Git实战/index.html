使用git fetch+rebase代替pull

通过这种的方法，可以更好地修改冲突；在自己环创的讨论群中了解到.
步骤：（git rebase 不会取回代码 要用git fetch先取回， git rebase 是合并代码。）

（1）首先用git fetch返回服务器上的代码
（2）首先用git rebase origin/master 合并
（3）如果发生冲突了会提示， 然后可以使用git diff查看冲突， 在手工改掉冲突， 在用git add ‘文件名’ 添加修改后文件，最后用git rebase —continue继续没完成的合并
（4）最后就可以用git push 更新到服务器上去。

Git Stash保存当前工作区间

场景：

我们往往会建一个自己的分支去修改和调试代码, 如果别人或者自己发现原有的分支上有个不得不修改的bug，我们往往会把完成一半的代码 commit提交到本地仓库，然后切换分支去修改bug，改好之后再切换回来。这样的话往往log上会有大量不必要的记录。其实如果我们不想提交完成一半或者不完善的代码，但是却不得不去修改一个紧急Bug，那么使用'git stash'就可以将你当前未提交到本地（和服务器）的代码推入到Git的栈中，这时候你的工作区间和上一次提交的内容是完全一样的，所以你可以放心的修 Bug，等到修完Bug，提交到服务器上后，再使用'git stash apply'将以前一半的工作应用回来

Git多人协作流程

公司使用git的多人协作方面，大家公用同一条分支，感觉这方面的使用还停留在我大二下半学期当中，几个人合作，就在master中，个人感觉冲突挺多的。

公用一个分支，推荐的使用流程：

一开始觉得这种提交方式很不科学，感觉并没有将git的强大功能体现出来。但是公司使用这种，自己也不得不屈服。直到今天才直到为什么要这么做。

首先在你修改完代码，想进行提交（commit）的时候，发现remote上，有很多新文件。这时，按我之前的经验是不管怎么样，我都得提交自己的代码，所以我会选择本地commit后，再进行pull，产生冲突，修改后，再进行push。这么做造成怎样的后果呢？如图：

这么做，后产生很多细小的分支，而且一些不必要的记录都会有，最后一个是：你的commit在前，其他人的在后面，而按道理来说，你的commit应该在后面才对，应为你commit的时候，别人已经提交，remote上已经有一份了。

所以，团队不推荐这么做，而是推荐上面的流程：

当改好想commit的时候，你发现remote已经有新的代码，将现在的代码stash起来，进行pull获取remote的所有代码后，再把stash的代码进行放出来，这时候会产生冲突，修改完成冲突后，跟自己修改好的代码一并提交commit+push。

可以发现你少了一次commit+push,把冲突的那次视为你的commit.那么细小的分支就不会这么多了。

带着疑问，上网查了Git多人协作的流程。看了之后，比较推崇的就是这篇了：
使用git和github进行协同开发流程 #7，对应衍生的辅助工具：Git Flow:

  • Git flow 開發流程
  • 开始实践git-flow

其实，上面的git flow的原理，就是上面推荐的那篇文章，所介绍的开发流程。下面简要减少流程，详细的看链接：

首先remote上有两个分支：master,develop:master存放最稳定的版本，最不活跃，不常更新。而develop有master分离，开发者将开发+测试好的代码放在此分支下。
接着,每个开发者都需要有着两个分支的副本，然后在develop分支下，衍生出属于自己的分支为A,此时在A分支上开发你需要做的模块的内容，开发完+测试后，将A分支合并到develop上，再有develop推送到remote上。

回想到之前，公司的一个测试项目，我就是这么的操作，结果还讲我使用图一的流程。原因：

提交代码的步骤建议调整下，否则很容易出现项目中代码被覆盖或丢失的情况

个人认为：只要操作妥当，是不会有丢失或者覆盖的情况发生的，因为git有一个强大的功能：回滚。最大的丢失在于merge产生冲突的时候。
